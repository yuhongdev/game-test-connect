# 阶段性汇报：前端 Auth (Register) 接口压力突刺测试结果

**测试目标：** `https://new.98ent.com` 注册 (Register) 接口承接高峰流量的极限摸底。
**测试工具：** Locust 分布式压力测试框架（采用 Step-Load 自动阶梯加压模式）
**单次最大并发量配置：** 1,000,000 测试账号

## 1. 测试工具与脚本升级总结 (前期修复)
针对由于服务器校验规则导致的压测失真问题，底层测试脚本现已全面升级：
- **突破服务器请求防御 (`code=3`)：** 初版生成的 17 位长测试账号触发了后端的 `username` 长度校验被拦截。我们已将海量测试账号全量缩进至 9 位以内（如 `st0000001`），成功使得真实请求打入数据库。
- **缓解云防火墙 WAF 防御 (`403 Forbidden`)：** 优化了虚拟用户的并发呼吸间隔 (Wait Time)，有效避开了被防火墙防刷策略 (Rate-Limiting) 误杀。
- **阶梯式压测法 (Step-Load) 上线：** 舍弃了全速全量倒灌的测试法，改用更科学的楼梯步进式——按既定“台阶”逐渐升高并发量，以便精准探明系统崩溃的“跳水点”。

## 2. 阶梯突刺测试：首发结果 (Burst Test)
在首轮的突刺测试中（每 60 秒递增 1,000 人并发），系统展现了明显的性能拐点：

- **顺畅期 (1,000 - 2,000 并发)：** 能够稳定吃下 600+ 次/秒 的吞吐量 (RPS)。数据库唯一性校验 (`dup` 识别) 工作正常，没有实质性故障。
- **崩溃红线 (逼近 3,000 并发)：** 当脚本继续推升时刻，由于短时间内涌入了高达 60,000 次/分钟的高频注册请求，系统发生**雪崩式阻断**。
  - **现象 1：** 接口失败率瞬间暴涨至 95% 以上 (`Fail=946` 等)。
  - **现象 2：** 高压诱发了底层死锁或延迟，导致本来已存在的账号被服务器错误地反馈为“成功注册” (`ok=340`)。
  - **现象 3：** 最为致命的是，**后端的管理控制台 (Backoffice) 也因此资源耗尽而发生瘫痪，完全无法加载。**

## 3. 结论概要
1. 在现有的架构下，**每秒 1,000 次以上的注册请求会直接击穿整个业务线**（前端 API 与 后台 Backoffice），证明系统目前缺乏针对超高并发的读写分离或资源隔离降级机制。
2. 突刺摸高目标已完成，下一阶段（Phase 2）不应继续尝试提高并发量，而应转向超低并发的“久置长测”（浸泡测试 / Soak Test），侦测服务器在日常中等流量下的深层资源漏洞。
